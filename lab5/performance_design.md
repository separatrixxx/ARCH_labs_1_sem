# Проектирование стратегии кеширования и rate limiting

**Система:** Profi — маркетплейс профессиональных услуг  
**Вариант:** 4 (Система маркетплейса услуг)  
**Архитектура:** PostgreSQL (users, orders) + MongoDB (services) + Redis (cache, rate limiting)

---

## 1. Анализ производительности

### 1.1 Hot paths (часто выполняемые операции)

| Endpoint | Частота | Характер нагрузки |
|----------|---------|-------------------|
| `GET /api/v1/services` | Очень высокая | Каждый пользователь просматривает каталог при входе |
| `GET /api/v1/services/{id}` | Высокая | Просмотр деталей услуги перед заказом |
| `GET /api/v1/users?login=...` | Высокая | Поиск пользователей при межсервисной коммуникации |
| `POST /api/v1/auth/login` | Средняя | Авторизация пользователей |
| `GET /api/v1/orders` | Средняя | Пользователи проверяют свои заказы |

### 1.2 Медленные операции

| Операция | Причина задержки | Оценка времени |
|----------|-----------------|----------------|
| `GET /api/v1/services` | Full-scan коллекции MongoDB | 10-50 мс |
| `GET /api/v1/users?first_name=&last_name=` | LIKE-запрос по PostgreSQL, даже с индексом | 5-30 мс |
| `POST /api/v1/orders/{id}/services` | HTTP-запрос к catalog-service для валидации | 20-100 мс |
| `GET /api/v1/services/{id}` | Запрос к MongoDB по ObjectId | 2-10 мс |

### 1.3 Требования к производительности

| Метрика | Целевое значение |
|---------|-----------------|
| Время отклика (p95) для GET-запросов | < 50 мс |
| Время отклика (p95) для POST-запросов | < 200 мс |
| Пропускная способность | > 500 RPS на сервис |
| Доступность | 99.9% |

## 2. Стратегия кеширования

### 2.1 Выбор данных для кеширования

| Данные | Частота чтения | Частота изменения | Решение |
|--------|---------------|-------------------|---------|
| Список услуг (`GET /services`) | Очень высокая | Низкая (услуги добавляются редко) | **Кешировать** |
| Услуга по ID (`GET /services/{id}`) | Высокая | Очень низкая | **Кешировать** |
| Пользователь по логину (`GET /users?login=`) | Высокая | Очень низкая (профиль редко меняется) | **Кешировать** |
| Заказы пользователя (`GET /orders`) | Средняя | Высокая (статусы меняются часто) | **Не кешировать** |

### 2.2 Стратегия для каждого типа данных

#### Список услуг — Cache-Aside (Lazy Loading)

```
Клиент → GET /api/v1/services
         ↓
    [Redis: GET "services:all"]
         ↓
    HIT → вернуть из кеша
    MISS → запросить MongoDB → сохранить в Redis → вернуть клиенту
```

**Обоснование выбора Cache-Aside:**
- Данные каталога читаются значительно чаще, чем обновляются (read-heavy workload)
- Приложение полностью контролирует, что и когда записывается в кеш
- При недоступности Redis система продолжает работать напрямую с MongoDB (graceful degradation)
- Нет необходимости в сложной синхронизации — достаточно инвалидации при записи

**Почему не другие стратегии:**
- **Read-Through**: требует отдельного cache-слоя между приложением и БД, избыточно для нашего масштаба
- **Write-Through**: синхронная запись в кеш при каждом обновлении добавит латентность на запись, а услуги обновляются редко
- **Write-Back**: асинхронная запись в БД создаёт риск потери данных, неоправданно для транзакционных операций

#### Пользователь по логину — Cache-Aside

```
Клиент → GET /api/v1/users?login=ivan.petrov
         ↓
    [Redis: GET "user:login:ivan.petrov"]
         ↓
    HIT → вернуть из кеша
    MISS → запросить PostgreSQL → сохранить в Redis → вернуть клиенту
```

**Обоснование:** поиск по логину — частая операция для межсервисной авторизации и валидации. Данные профиля меняются крайне редко.

#### Услуга по ID — Cache-Aside

```
Клиент → GET /api/v1/services/{id}
         ↓
    [Redis: GET "service:{id}"]
         ↓
    HIT → вернуть из кеша
    MISS → запросить MongoDB → сохранить в Redis → вернуть клиенту
```

### 2.3 TTL (Time To Live)

| Ключ кеша | TTL | Обоснование |
|-----------|-----|-------------|
| `services:all` | 60 сек | Каталог обновляется редко, но пользователи должны видеть новые услуги в разумный срок |
| `service:{id}` | 120 сек | Конкретная услуга ещё стабильнее, чем весь каталог |
| `user:login:{login}` | 30 сек | Пользовательские данные могут меняться (при реализации обновления профиля) |

### 2.4 Стратегия инвалидации кеша

| Событие | Действие |
|---------|----------|
| `POST /api/v1/services` (создание услуги) | `DEL services:all` — полный сброс кеша каталога |
| Обновление услуги (будущее) | `DEL services:all` + `DEL service:{id}` |
| Обновление профиля (будущее) | `DEL user:login:{login}` |

**Принцип:** при модификации данных немедленно удаляем соответствующий ключ из кеша. Следующий запрос на чтение обнаружит cache miss и загрузит актуальные данные из базы.

## 3. Стратегия rate limiting

### 3.1 Endpoints, требующие rate limiting

| Endpoint | Причина | Угрозы |
|----------|---------|--------|
| `POST /api/v1/auth/login` | Предотвращение brute-force атак | Подбор пароля, credential stuffing |
| `POST /api/v1/auth/register` | Защита от массовой регистрации ботов | Spam accounts, resource exhaustion |
| `POST /api/v1/orders` | Защита от злоупотреблений | Создание фиктивных заказов |

### 3.2 Выбор алгоритма

#### Для auth-эндпоинтов — Fixed Window Counter

```
Окно: 60 секунд
Ключ: "rl:{endpoint}:{client_ip}"

Запрос → INCR ключа → если count == 1, EXPIRE 60 сек
         ↓
    count <= limit → пропустить
    count > limit  → HTTP 429 Too Many Requests
```

**Обоснование выбора Fixed Window Counter:**
- Простота реализации: одна атомарная операция INCR в Redis
- Минимальное потребление памяти: один ключ на IP + endpoint + окно
- Автоматическая очистка через TTL Redis
- Достаточная точность для защиты auth-эндпоинтов

**Сравнение с альтернативами:**

| Алгоритм | Плюсы | Минусы | Подходит? |
|----------|-------|--------|-----------|
| **Fixed Window Counter** | Простой, O(1) memory, атомарный INCR | Всплеск на границе окна (до 2x лимита) | **Да** — простота важнее точности для auth |
| Token Bucket | Гладкий трафик, настраиваемый burst | Сложнее реализовать атомарно в Redis | Нет — избыточен |
| Leaking Bucket | Строго фиксированная скорость | Не допускает burst вообще, плохой UX | Нет — слишком строгий |
| Sliding Window Log | Точный подсчёт, нет проблемы границ | O(N) memory на каждый IP | Нет — дорого по памяти |
| Sliding Window Counter | Хороший компромисс точности и памяти | Сложнее в реализации | Возможно, для будущих версий |

### 3.3 Лимиты

| Endpoint | Лимит | Окно | Обоснование |
|----------|-------|------|-------------|
| `POST /api/v1/auth/login` | 10 запросов | 60 сек | Защита от brute-force; обычный пользователь логинится 1-2 раза |
| `POST /api/v1/auth/register` | 5 запросов | 60 сек | Строже: регистрация — разовое действие |

### 3.4 HTTP-заголовки rate limiting

При каждом ответе на rate-limited эндпоинтах возвращаются заголовки:

```
X-RateLimit-Limit: 10          # максимум запросов в окне
X-RateLimit-Remaining: 7       # осталось запросов
X-RateLimit-Reset: 45          # секунд до сброса окна
```

При превышении лимита:

```
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
X-RateLimit-Limit: 10
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 23

{"error": "Too many requests. Try again later."}
```

## 4. Реализация

### 4.1 Архитектура

```
                    ┌──────────┐
                    │  nginx   │ :8080  (reverse proxy, CORS)
                    └────┬─────┘
            ┌────────────┼────────────┐
            ▼            ▼            ▼
     ┌────────────┐ ┌──────────┐ ┌──────────┐
     │user-service│ │ catalog  │ │  order   │
     │   :8081    │ │ service  │ │ service  │
     │            │ │  :8082   │ │  :8083   │
     └──┬────┬────┘ └──┬──┬───┘ └──┬──┬────┘
        │    │         │  │        │  │
        ▼    ▼         ▼  ▼        ▼  ▼
  ┌────────┐ ┌─────┐ ┌─────┐ ┌────────┐
  │Postgres│ │Redis│ │Mongo│ │Postgres│
  └────────┘ └─────┘ └─────┘ └────────┘
```

Redis используется всеми тремя сервисами как единый кеш- и rate-limit-слой.

### 4.2 Компонент RedisCache

Реализован как userver-компонент `RedisCache` (файлы `src/shared/redis_cache.hpp`, `redis_cache.cpp`):

- **Подключение:** hiredis (синхронный C-клиент) с auto-reconnect
- **Thread safety:** `std::mutex` защищает все операции с `redisContext`
- **Fail-open:** при недоступности Redis запросы проходят напрямую к БД (graceful degradation)

Методы:
- `Get(key)` / `Set(key, value, ttl)` / `Del(key)` — базовые операции кеша
- `CheckRateLimit(key, max, window)` — Fixed Window Counter (INCR + EXPIRE + TTL)

### 4.3 Кеширование (3 эндпоинта)

| Endpoint | Cache key | TTL | Инвалидация |
|----------|-----------|-----|-------------|
| `GET /api/v1/services` | `services:all` | 60 с | `POST /api/v1/services` → `DEL services:all` |
| `GET /api/v1/services/{id}` | `service:{id}` | 120 с | По TTL |
| `GET /api/v1/users?login=...` | `user:login:{login}` | 30 с | По TTL |

Заголовок `X-Cache: HIT/MISS` позволяет отслеживать попадания в кеш.

### 4.4 Rate limiting (2 эндпоинта)

| Endpoint | Алгоритм | Ключ | Лимит |
|----------|----------|------|-------|
| `POST /api/v1/auth/login` | Fixed Window | `rl:login:{ip}` | 10/мин |
| `POST /api/v1/auth/register` | Fixed Window | `rl:register:{ip}` | 5/мин |

## 5. Анализ влияния на производительность

### 5.1 Улучшения от кеширования

| Метрика | Без кеша | С кешем (hit) | Улучшение |
|---------|----------|---------------|-----------|
| `GET /services` latency | 10-50 мс (MongoDB) | < 1 мс (Redis) | ~10-50x |
| `GET /services/{id}` latency | 2-10 мс (MongoDB) | < 1 мс (Redis) | ~2-10x |
| `GET /users?login=` latency | 5-15 мс (PostgreSQL) | < 1 мс (Redis) | ~5-15x |
| Нагрузка на MongoDB | 100% запросов | ~5-10% запросов | -90% |
| Нагрузка на PostgreSQL | 100% запросов | ~10-20% запросов | -80% |

### 5.2 Улучшения от rate limiting

- **Безопасность:** защита от brute-force атак на авторизацию
- **Стабильность:** предотвращение перегрузки сервиса массовыми запросами
- **Справедливость:** один пользователь/IP не может монополизировать ресурсы

### 5.3 Метрики для мониторинга

| Метрика | Описание | Как измерить |
|---------|----------|-------------|
| **Cache hit rate** | % запросов, обслуженных из кеша | Подсчёт `X-Cache: HIT` vs `MISS` в логах nginx |
| **Cache latency** | Время ответа Redis GET | Логирование в RedisCache::Get() |
| **Rate limit rejections** | Количество HTTP 429 ответов | Подсчёт 429 статусов в логах nginx |
| **Redis memory usage** | Объём памяти Redis | `redis-cli INFO memory` |
| **Redis connection errors** | Ошибки подключения к Redis | Логирование в RedisCache::EnsureConnected() |

### 5.4 Измерение эффективности кеширования (Hit Rate)

**Hit Rate = количество HIT / (количество HIT + количество MISS) * 100%**

Способы измерения:

1. **Через HTTP-заголовки:** каждый кешированный ответ содержит `X-Cache: HIT` или `X-Cache: MISS`. Анализ логов nginx:
   ```bash
   # Общий hit rate
   grep "X-Cache" access.log | awk '{print $NF}' | sort | uniq -c

   # Hit rate по эндпоинту
   grep "/api/v1/services" access.log | grep -c "X-Cache: HIT"
   ```

2. **Через Redis INFO:**
   ```bash
   redis-cli INFO stats | grep keyspace
   # keyspace_hits: 12345
   # keyspace_misses: 678
   # Hit rate = 12345 / (12345 + 678) = 94.8%
   ```

3. **Через Redis MONITOR** (для отладки, не для продакшена):
   ```bash
   redis-cli MONITOR | grep -E "GET|SET"
   ```

**Целевой hit rate:** > 80% для `services:all` (высокочастотный эндпоинт).
