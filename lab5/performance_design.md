# Проектирование стратегии кеширования и rate limiting

**Система:** Profi — маркетплейс профессиональных услуг  
**Вариант:** 4 (Система маркетплейса услуг)  
**Архитектура:** PostgreSQL (users, orders) + MongoDB (services) + Redis (cache, rate limiting)

---

## 1. Анализ производительности

### 1.1 Hot paths

Прежде чем лезть оптимизировать, надо понять, что вообще тормозит и что вызывается чаще всего. Прошёлся по эндпоинтам:

| Endpoint | Частота | Характер нагрузки |
|----------|---------|-------------------|
| `GET /api/v1/services` | Очень высокая | Каждый пользователь просматривает каталог при входе |
| `GET /api/v1/services/{id}` | Высокая | Просмотр деталей услуги перед заказом |
| `GET /api/v1/users?login=...` | Высокая | Поиск пользователей, межсервисная коммуникация |
| `POST /api/v1/auth/login` | Средняя | Авторизация |
| `GET /api/v1/orders` | Средняя | Пользователи проверяют свои заказы |

### 1.2 Медленные операции

| Операция | Причина задержки | Оценка времени |
|----------|-----------------|----------------|
| `GET /api/v1/services` | Full-scan коллекции MongoDB | 10-50 мс |
| `GET /api/v1/users?first_name=&last_name=` | LIKE-запрос по PostgreSQL, даже с индексом | 5-30 мс |
| `POST /api/v1/orders/{id}/services` | HTTP-запрос к catalog-service для валидации | 20-100 мс |
| `GET /api/v1/services/{id}` | Запрос к MongoDB по ObjectId | 2-10 мс |

Каталог услуг — самое узкое место. Его листают постоянно, а каждый раз идёт full-scan по MongoDB (у нас пока не так много данных, но с ростом это станет проблемой).

### 1.3 Требования к производительности

| Метрика | Целевое значение |
|---------|-----------------|
| Время отклика (p95) для GET | < 50 мс |
| Время отклика (p95) для POST | < 200 мс |
| Пропускная способность | > 500 RPS на сервис |
| Доступность | 99.9% |

## 2. Стратегия кеширования

### 2.1 Что кешировать, а что нет

Не всё подряд стоит пихать в кеш. Смысл есть только если данные читаются часто, а меняются редко:

| Данные | Частота чтения | Частота изменения | Решение |
|--------|---------------|-------------------|---------|
| Список услуг (`GET /services`) | Очень высокая | Низкая (услуги добавляются редко) | **Кешировать** |
| Услуга по ID (`GET /services/{id}`) | Высокая | Очень низкая | **Кешировать** |
| Пользователь по логину (`GET /users?login=`) | Высокая | Очень низкая (профиль редко меняется) | **Кешировать** |
| Заказы пользователя (`GET /orders`) | Средняя | Высокая (статусы постоянно меняются) | **Не кешировать** |

Заказы — осознанное решение не кешировать. Там статусы меняются часто (pending → confirmed → completed), и кеш будет постоянно протухать. Проще читать из базы напрямую.

### 2.2 Стратегия: Cache-Aside (Lazy Loading)

Для всех трёх кешируемых эндпоинтов выбрал одну стратегию — Cache-Aside. Логика простая:

```
Клиент → GET /api/v1/services
         ↓
    [Redis: GET "services:all"]
         ↓
    HIT → вернуть из кеша
    MISS → запросить MongoDB → сохранить в Redis → вернуть клиенту
```

Почему именно Cache-Aside, а не что-то другое:

- **Read-Through** — требует отдельного прозрачного cache-слоя между приложением и БД. Для нашего масштаба это лишнее усложнение.
- **Write-Through** — при каждой записи синхронно обновляем кеш. Но услуги создаются редко, а латентность на запись увеличится. Смысла нет.
- **Write-Back** — запись сначала в кеш, потом асинхронно в БД. Опасная штука — если Redis упадёт до сброса в БД, данные потеряются. Для нас неприемлемо.

Cache-Aside хорош тем, что при падении Redis ничего страшного не случится — просто запросы пойдут напрямую в базу, как раньше. Graceful degradation из коробки.

Для пользователя по логину и услуги по ID всё то же самое:

```
Клиент → GET /api/v1/users?login=ivan.petrov
         ↓
    [Redis: GET "user:login:ivan.petrov"]
         ↓
    HIT → вернуть из кеша
    MISS → запросить PostgreSQL → сохранить в Redis → вернуть клиенту
```

### 2.3 TTL

| Ключ кеша | TTL | Почему так |
|-----------|-----|-----------|
| `services:all` | 60 сек | Каталог обновляется нечасто, но хочется чтобы новые услуги появлялись в разумный срок |
| `service:{id}` | 120 сек | Конкретная услуга меняется ещё реже, чем весь каталог — можно кешировать дольше |
| `user:login:{login}` | 30 сек | Покороче, потому что в будущем может появиться редактирование профиля |

### 2.4 Инвалидация

| Событие | Действие |
|---------|----------|
| `POST /api/v1/services` (создание услуги) | `DEL services:all` — сбрасываем кеш каталога целиком |
| Обновление услуги (пока не реализовано) | `DEL services:all` + `DEL service:{id}` |
| Обновление профиля (пока не реализовано) | `DEL user:login:{login}` |

Принцип простой: при любом изменении данных — удаляем ключ из кеша. Следующий GET-запрос увидит miss и подтянет свежие данные из базы. Не пытаемся обновлять кеш напрямую — так проще и надёжнее.

## 3. Стратегия rate limiting

### 3.1 Какие эндпоинты ограничивать

Rate limiting нужен не везде — в первую очередь на эндпоинтах, которые можно абьюзить:

| Endpoint | Причина | Угрозы |
|----------|---------|--------|
| `POST /api/v1/auth/login` | Brute-force атаки | Подбор пароля, credential stuffing |
| `POST /api/v1/auth/register` | Массовая регистрация ботов | Спам-аккаунты |
| `POST /api/v1/orders` | Злоупотребление созданием заказов | Фиктивные заказы |

Пока реализовал только для login и register — это самые критичные. Заказы можно будет добавить потом.

### 3.2 Выбор алгоритма: Fixed Window Counter

Для auth-эндпоинтов взял Fixed Window Counter. Работает так:

```
Окно: 60 секунд
Ключ: "rl:{endpoint}:{client_ip}"

Запрос → INCR ключа → если count == 1, EXPIRE 60 сек
         ↓
    count <= limit → пропустить
    count > limit  → HTTP 429 Too Many Requests
```

Рассматривал и другие варианты:

| Алгоритм | Плюсы | Минусы | Подходит? |
|----------|-------|--------|-----------|
| **Fixed Window Counter** | Простой, O(1) memory, атомарный INCR | Всплеск на границе окна (до 2x лимита) | **Да** |
| Token Bucket | Гладкий трафик, настраиваемый burst | Сложнее реализовать атомарно | Избыточен |
| Leaking Bucket | Строго фиксированная скорость | Не допускает burst, плохой UX | Слишком жёсткий |
| Sliding Window Log | Точный подсчёт, нет проблемы границ | O(N) memory на каждый IP | Дорого по памяти |
| Sliding Window Counter | Хороший компромисс | Сложнее в реализации | Может быть, в будущем |

Да, у Fixed Window есть проблема с границей окна — теоретически можно отправить до 2x лимита, если попасть на стык двух окон. Но для auth-эндпоинтов это некритично: даже если кто-то пошлёт 20 запросов вместо 10, это всё равно не полноценный brute-force. А вот простота реализации — один INCR в Redis — перевесила.

### 3.3 Лимиты

| Endpoint | Лимит | Окно | Почему |
|----------|-------|------|--------|
| `POST /api/v1/auth/login` | 10 запросов | 60 сек | Нормальный пользователь логинится 1-2 раза, 10 — с большим запасом |
| `POST /api/v1/auth/register` | 5 запросов | 60 сек | Строже: регистрация — это разовое действие, 5 раз в минуту — уже подозрительно |

### 3.4 HTTP-заголовки

В каждом ответе на rate-limited эндпоинт отдаём информацию о лимитах:

```
X-RateLimit-Limit: 10          # максимум запросов в окне
X-RateLimit-Remaining: 7       # осталось
X-RateLimit-Reset: 45          # секунд до сброса
```

Если лимит превышен:

```
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
X-RateLimit-Limit: 10
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 23

{"error": "Too many requests. Try again later."}
```

## 4. Реализация

### 4.1 Архитектура

```
                    ┌──────────┐
                    │  nginx   │ :8080  (reverse proxy, CORS)
                    └────┬─────┘
            ┌────────────┼────────────┐
            ▼            ▼            ▼
     ┌────────────┐ ┌──────────┐ ┌──────────┐
     │user-service│ │ catalog  │ │  order   │
     │   :8081    │ │ service  │ │ service  │
     │            │ │  :8082   │ │  :8083   │
     └──┬────┬────┘ └──┬──┬───┘ └──┬──┬────┘
        │    │         │  │        │  │
        ▼    ▼         ▼  ▼        ▼  ▼
  ┌────────┐ ┌─────┐ ┌─────┐ ┌────────┐
  │Postgres│ │Redis│ │Mongo│ │Postgres│
  └────────┘ └─────┘ └─────┘ └────────┘
```

Redis один на всех — и кеш, и rate limiting через него.

### 4.2 Компонент RedisCache

Написал userver-компонент `RedisCache` (лежит в `src/shared/redis_cache.hpp` и `redis_cache.cpp`). Под капотом hiredis — синхронный C-клиент для Redis. Не самый изящный вариант, но для наших целей хватает.

Что внутри:
- `Get(key)` / `Set(key, value, ttl)` / `Del(key)` — стандартные операции
- `CheckRateLimit(key, max, window)` — Fixed Window Counter через INCR + EXPIRE + TTL
- `std::mutex` на все операции с `redisContext` для потокобезопасности
- Fail-open: если Redis недоступен, запрос идёт к БД напрямую, а не падает с ошибкой

### 4.3 Кеширование (3 эндпоинта)

| Endpoint | Cache key | TTL | Инвалидация |
|----------|-----------|-----|-------------|
| `GET /api/v1/services` | `services:all` | 60 с | `POST /api/v1/services` → `DEL services:all` |
| `GET /api/v1/services/{id}` | `service:{id}` | 120 с | По TTL |
| `GET /api/v1/users?login=...` | `user:login:{login}` | 30 с | По TTL |

В каждый ответ добавляется заголовок `X-Cache: HIT` или `MISS` — удобно для отладки и для подсчёта hit rate.

### 4.4 Rate limiting (2 эндпоинта)

| Endpoint | Алгоритм | Ключ | Лимит |
|----------|----------|------|-------|
| `POST /api/v1/auth/login` | Fixed Window | `rl:login:{ip}` | 10/мин |
| `POST /api/v1/auth/register` | Fixed Window | `rl:register:{ip}` | 5/мин |

## 5. Анализ влияния на производительность

### 5.1 Что дало кеширование

| Метрика | Без кеша | С кешем (hit) | Разница |
|---------|----------|---------------|---------|
| `GET /services` latency | 10-50 мс (MongoDB) | < 1 мс (Redis) | ~10-50x |
| `GET /services/{id}` latency | 2-10 мс (MongoDB) | < 1 мс (Redis) | ~2-10x |
| `GET /users?login=` latency | 5-15 мс (PostgreSQL) | < 1 мс (Redis) | ~5-15x |
| Нагрузка на MongoDB | 100% запросов | ~5-10% запросов | -90% |
| Нагрузка на PostgreSQL | 100% запросов | ~10-20% запросов | -80% |

Самый ощутимый эффект — на каталоге услуг. Он запрашивается чаще всего, и разница между full-scan MongoDB и GET из Redis огромная.

### 5.2 Что дал rate limiting

С точки зрения производительности rate limiting — это не про скорость, а про стабильность и безопасность:
- Защита от brute-force на логин
- Один IP не может забить сервис массовыми запросами
- При DDoS-подобной нагрузке сервис остаётся доступным для нормальных пользователей

### 5.3 Метрики для мониторинга

| Метрика | Как измерить |
|---------|-------------|
| Cache hit rate | Считаем `X-Cache: HIT` vs `MISS` в логах nginx |
| Cache latency | Логирование в RedisCache::Get() |
| Rate limit rejections | Количество 429-х в логах nginx |
| Redis memory | `redis-cli INFO memory` |
| Redis connection errors | Логирование в RedisCache::EnsureConnected() |

### 5.4 Как считать hit rate

**Hit Rate = HIT / (HIT + MISS) * 100%**

Три способа померить:

1. По HTTP-заголовкам — у нас каждый кешированный ответ содержит `X-Cache`, можно грепнуть логи:
   ```bash
   grep "X-Cache" access.log | awk '{print $NF}' | sort | uniq -c
   ```

2. Через Redis INFO — там есть глобальная статистика:
   ```bash
   redis-cli INFO stats | grep keyspace
   # keyspace_hits: 12345
   # keyspace_misses: 678
   # Hit rate = 12345 / (12345 + 678) = 94.8%
   ```

3. Через `redis-cli MONITOR` — видно каждую операцию в реальном времени. Но это только для отладки, в проде включать нельзя — просаживает производительность.

Для нашего каталога (`services:all`) целевой hit rate > 80%. С TTL в 60 секунд и высокой частотой чтения это достижимо.
